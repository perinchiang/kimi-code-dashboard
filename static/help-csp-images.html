Kimi Web 外网访问图片不显示

Content-Security-Policy 导致的图床/外部图片加载失败修复指南

现象

通过 SakuraFrp、ngrok 等内网穿透访问 Kimi Web 时，消息里的图片显示裂图或 Image failed to load。

注意（kimi-code ≥ 0.25）：CSP 按服务端绑定地址决定是否下发，与你怎么访问无关。只要 Kimi Web 是用 --host（0.0.0.0）启动的，即使是在本机打开 http://127.0.0.1:5494，图床外链图片同样会裂；只有服务端绑定 127.0.0.1 时才完全不下发 CSP。

另外，AI 发送的本地文件路径图片（如 C:/Users/... 或 file://）在外网也是看不到的，因为路径指向的是 AI 所在电脑的本地磁盘，不是你的电脑。

原因

Kimi Code Web 在非回环绑定（--host / 0.0.0.0）时会在 HTTP 响应头中下发严格的 CSP 策略（0.28 实测）：

Content-Security-Policy: default-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data: blob:; font-src 'self' data:; form-action 'self'; base-uri 'none'; frame-ancestors 'self'

其中 img-src 'self' data: blob: 表示浏览器只允许加载同域名、Base64 内嵌和 blob 图片。任何外部图床 URL（如 https://pub-xxx.r2.dev/...、https://image.pathos.dpdns.org/...）都会被浏览器直接拦截；Base64 内嵌图（data:）不受影响。

在浏览器 DevTools Console 中可以看到类似报错：

Loading the image 'https://...' violates the following Content Security Policy directive: "img-src 'self' data: blob:".

外网访问必须搭配图床

即使 CSP 头已经修复，外网访问时 AI 发送的图片也必须是图床 URL，不能是本地文件路径。

图片来源 本机 127.0.0.1 外网穿透后
本地文件路径 C:/... / file:// 可见 不可见（指向你的本地磁盘）
Base64 内嵌 data:image/png;base64,... 可见 可见（需 CSP 允许 data:）
图床 URL https://image.xxx/... 可见 可见（需 CSP 允许外部图片）

推荐做法

  1. 在 Dashboard 设置 → MCP 图床配置 里填入 R2 / S3 / MinIO / OSS 凭证
  2. AI 生成图片后，会自动上传图床，再发送图床外链给你
  3. 配合本页 CSP 修复方案，外网访问时图片即可正常显示
注意：如果不配置图床，AI 只能发送 Base64 内嵌图或本地路径图；本地路径图在外网穿透环境下对方是看不到的。

解决方案对照表

访问方式 能否改 CSP 头 推荐做法
本机 127.0.0.1 无需处理 仅当服务端绑定 127.0.0.1 时无 CSP；若服务端绑定了 0.0.0.0，本机访问同样被拦，需走方案 A/C
SakuraFrp / ngrok 不能 本机加 Caddy 反向代理（见方案 A）
Cloudflare Tunnel 能 Cloudflare 面板加 Transform Rule（见方案 B）
自建 Nginx / Caddy 反代 能 在反代配置里改响应头（见方案 C）

方案 A：SakuraFrp + 本地 Caddy

在 Kimi Web 和 SakuraFrp 之间加一层 Caddy，让 Caddy 修改响应头。

1. 安装 Caddy

Windows 推荐用 WinGet：

winget install Caddy.Caddy

macOS 用 Homebrew：

brew install caddy

2. 创建 Caddyfile

新建 Caddyfile（路径任意，例如 ~/.kimi-code/pwa-proxy/Caddyfile）：

:8088 {
    # PWA 静态资源（可选）
    @pwaStatic path / /manifest.json /icon-192.png /icon-512.png /apple-touch-icon.png /favicon.ico
    handle @pwaStatic {
        root * ~/.kimi-code/pwa-proxy/static
        file_server
    }

    # 其他请求转发到 Kimi Web
    handle {
        reverse_proxy http://127.0.0.1:5494 {
            # 用 header_down 直接改写上游响应头（注意：header -CSP 是延迟删除，
            # 会把后面 header 指令新设置的 CSP 一并删掉，不要那样写）
            header_down Content-Security-Policy "default-src 'self'; img-src * data: blob:; font-src 'self' data:; script-src 'self'; connect-src 'self' ws: wss: http: https:; style-src 'self' 'unsafe-inline'; form-action 'self'; base-uri 'none'; frame-ancestors 'self';"
        }
    }
}

3. 启动 Caddy

caddy run --config /path/to/Caddyfile
Windows 开机自启建议：不要用 WinGet Links 里的 shim（...\WinGet\Links\caddy.exe）配合 caddy start 做自启，登录时序下不稳定。推荐在「启动」文件夹放一个 VBS，直接调用真实路径的 caddy.exe run（真实路径形如 %LOCALAPPDATA%\Microsoft\WinGet\Packages\CaddyServer.Caddy_...\caddy.exe），隐藏窗口前台运行最可靠。

4. 修改 SakuraFrp 隧道

把隧道的本地端口从 5494 改成 8088，其他保持不变。

之后通过 frp 域名访问时，流量会先经过 Caddy，CSP 头被移除，外部图片即可正常显示。

方案 B：Cloudflare Tunnel

Cloudflare Tunnel 支持在云端修改响应头，不需要本地 Caddy。

  1. 登录 Cloudflare 控制台 → 你的域名 → Rules → Transform Rules
  2. 创建 Modify Response Header 规则
  3. 匹配条件：Hostname equals 你的 tunnel 域名
  4. 操作：Remove header Content-Security-Policy
  5. 保存并部署
提示：也可以把 Remove 改成 Set，值为宽松的 CSP 策略，和方案 A 中一致。

方案 C：自建 Nginx 反代

如果你前面已经有一层 Nginx，直接加响应头即可：

server {
    listen 443 ssl;
    server_name kimi.example.com;

    location / {
        proxy_pass http://127.0.0.1:5494;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;

        # 移除 Kimi Web 的 CSP
        proxy_hide_header Content-Security-Policy;
        add_header Content-Security-Policy "default-src 'self'; img-src * data: blob:; font-src 'self' data:; script-src 'self' 'unsafe-inline'; connect-src 'self' ws: wss: http: https:; style-src 'self' 'unsafe-inline';" always;
    }
}

验证是否修复

  1. 按 F12 打开浏览器 DevTools → Network
  2. 刷新页面，点击任意一个 HTML 请求（如 /app/）
  3. 查看 Response Headers
  4. 如果 Content-Security-Policy 已消失或被替换成宽松策略，即修复成功
